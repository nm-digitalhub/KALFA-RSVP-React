### חקירת "דפים כפולים" — ממצאי שלושה סוכני חקירה בלתי-תלויים (06.07.2026)

**מסקנה חד-משמעית: אין דפים כפולים של אישורי הגעה.** שלושה סוכנים (מסלולים/קוד, תנאי-נראות, מפקד DB חי) הגיעו לאותה תשובה מכיוונים שונים:

1. **מימוש יחיד:** קיים עמוד RSVP ציבורי אחד בלבד — `/r/[token]` — והרכיב `rsvp-form.tsx` הוא הקובץ היחיד בכל הריפו שמרנדר טופס אישור הגעה (grep ל-`meal_pref`/"האם תגיעו" — קובץ אחד). ‏`/g/[token]` הוא redirect למתנה (ללא HTML), ‏`/join/[token]` הוא הצטרפות ארגון. היסטוריית git נקייה — אף עמוד RSVP לא נמחק/נותב מחדש, אין מימוש מקביל באף branch, והשדה היה מותנה-"מגיע/ה" עוד מהקומיט הראשון. כל קישורי הוואטסאפ מצביעים על `/r/<token>` בלבד.
2. **ייחודיות טוקנים ב-DB:** אילוץ UNIQUE על `guests.rsvp_token` ‏+ ‏0 טוקנים משותפים — אין "עמוד כפול" דרך טוקן.
3. **טבלת ההחלטה המלאה** (מתי אין שדה תפריט):

| מצב | טופס מוצג? | שדה תפריט? |
|---|---|---|
| אורח במצב "מגיע/ה" (בחר עכשיו או תשובה שמורה) | ✔ | ✔ (גם ב-HTML מהשרת) |
| אורח שטרם ענה (pending) — אף כפתור לא בחור | ✔ | ✖ עד לחיצה על "מגיע/ה" |
| "לא מגיע/ה" / "אולי" | ✔ | ✖ |
| דדליין עבר או תאריך האירוע עבר (`can_respond=false`) | ✖ ("המועד לאישור הגעה חלף") | ✖ |
| טוקן שגוי/בוטל, אירוע טיוטה/סגור, rate-limit | ✖ (שגיאה גנרית) | ✖ |
| כשל hydration (נדיר, סביבתי) | ✔ | ✖ — הלחיצה על "מגיע/ה" לא מגיבה |

4. **המפקד החי מסביר את מה שראית בפועל:** בכל האירועים הפעילים כרגע — **כל 7 האורחים במצב pending** (איש טרם ענה), וב-2 מתוך 4 האירועים הפעילים הדדליין/תאריך כבר עבר ואין טופס בכלל. כלומר **כל עמוד RSVP חי שנפתח היום מציג "אין שדה תפריט" בכניסה — בעיצוב, לא בבאג.**
5. **פער מבני אמיתי שהחקירה חשפה (רלוונטי להמשך):** מסלולי הוואטסאפ (כפתור אישור מהיר + תשובת headcount) לעולם אינם אוספים העדפת תפריט — `submitRsvp` נקרא בלי `meal`, ו-headcount כותב כמויות בלבד. אורח שיאשר דרך וואטסאפ יגיע תמיד עם `meal_pref=null`. כרגע 0 מקרים חיים, אבל זה יתרחש ברגע שקמפיין ירוץ. שווה החלטת מוצר (למשל: שאלת continuation בוואטסאפ או השלמה בקישור).

## חלק ב׳ — תכנון: מתג פר-אירוע להצגת/הסתרת שדה העדפת התפריט

### מטרה
לאפשר לבעל האירוע לקבוע, ברמת האירוע, האם טופס אישור ההגעה הציבורי יציג את שדה "העדפת תפריט" (יש אירועים — ברית/מסיבה — שבהם השדה מיותר ומבלבל).

### עיצוב מוצע (מינימלי, עקבי עם דפוסי הפרויקט)

1. **DB — מיגרציה חדשה**
   - `alter table public.events add column if not exists show_meal_pref boolean not null default true;` ‏+ ‏comment.
   - ברירת מחדל `true` = שימור ההתנהגות לכל האירועים הקיימים.
   - עדכון `get_rsvp_by_token`: הוספת `'show_meal_pref', e.show_meal_pref` ל-jsonb של ה-event (שינוי שורה אחת, באותה משמעת byte-diff כמו rsvp_note).
   - **הקשחת `submit_rsvp` (הגנה בצד שרת, לא רק UI):** כאשר `show_meal_pref = false` ⇒ ‏`_meal_n := null` גם במצב attending — לקוח ישן/זדוני לא יוכל לכתוב העדפה כשהמתג כבוי.

2. **טיפוסים** — ‏`types.ts` (events Row/Insert/Update) + ‏`RsvpEventInfo.show_meal_pref: boolean` ב-`src/lib/data/rsvp.ts`.

3. **UI בעלים** — בטופס "עריכת פרטי האירוע" (`event-edit-form`), checkbox בסגנון הקיים:
   - תווית: **"איסוף העדפת תפריט מהאורחים"** + הסבר קצר: "כשמופעל, טופס אישור ההגעה יציג לאורחים שמאשרים הגעה שדה חופשי להעדפת תפריט."
   - מיקום: קבוצת שדות הטופס הציבורי (ליד קישור מתנה/תמונת הזמנה). **לא נעול אחרי פרסום** — ניתן לכיבוי/הדלקה בכל שלב (בשונה מתאריך/דדליין).
   - עדכון ה-server action של עריכת אירוע + סכמת ה-Zod שלו (boolean).

4. **טופס ציבורי** — ‏`rsvp-form.tsx`: התנאי הופך ל-`{attending && event.show_meal_pref !== false ? ... : null}`.
   - ההשוואה `!== false` (ולא truthy) מגינה על חלון פריסה שבו הקוד חדש וה-DB עדיין ישן (מפתח חסר ⇒ ברירת מחדל להציג).

5. **בדיקות**
   - עדכון/הוספה ב-vitest: ולידציית עריכת אירוע כוללת את הדגל; ‏mapping ‏RsvpEventInfo.
   - tripwire בסגנון `rsvp-privacy.test.ts`: המיגרציה מוסיפה את המפתח ל-get_rsvp_by_token ואת ההגנה ל-submit_rsvp.
   - E2E ידני: מתג כבוי ⇒ אורח מאשר הגעה בלי שדה תפריט; הגשה עם `meal_pref` מזויף ב-FormData לא כותבת ל-DB.

6. **פריסה** — אותה משמעת: מיגרציה ל-DB החי ← ‏deploy ← אימות בדפדפן. אין השפעה על נתונים קיימים; כיבוי המתג לא מוחק ערכים שכבר נאספו.

### שיקולים וסיכונים
- **סיכון נמוך.** שינוי אדיטיבי; ברירת מחדל משמרת התנהגות; דגל נקרא דרך אותו RPC קיים.
- **תלות מומלצת ב-M-1:** אם מוסיפים מתג לאיסוף — ראוי באותה חבילה להוסיף גם **תצוגת ההעדפות לבעלים** (עמוד מוזמן + עמודה/ייצוא), אחרת המתג שולט על איסוף נתון שאיש אינו רואה.
- **חלופה ארכיטקטונית שנשקלה ונדחתה (לשלב זה):** הפיכת "העדפת תפריט" לשאלה מובנית במנגנון `event_questions` הקיים (הפעלה/כיבוי דרך ניהול שאלות). יתרון: אחידות; חיסרון: מיגרציית נתונים ושבירת ה-flow/הדוחות הקיימים סביב `meal_pref`. אפשר לחזור לזה כשמנגנון השאלות יקבל UI ניהול מלא.
- **שאלת מוצר פתוחה:** האם המתג צריך להיות לפי סוג אירוע כברירת מחדל (למשל כבוי בברית/בריתה)? מוצע: לא בשלב זה — ברירת מחדל true אחידה, הבעלים מחליט.
