# `/api/sdk-auth` — מצב ההכנה ותוכנית יישום

> נכתב 2026-07-21. כל עובדה כאן אומתה מול ה-DB החי, מול הקוד, או מול התיעוד הרשמי
> של Voximplant דרך `voximplant.com/api/v2/getDoc`. מה שלא אומת מסומן במפורש.

## 1. מה זה ולמה זה חוסם

האפליקציה (KALFA-ELEVENLABS) מתחברת ל-Voximplant Android SDK כדי שנציג אנושי יוכל
להאזין לשיחת AI חיה ולהשתלט עליה. ההתחברות היא חילופי מפתח חד-פעמי, והשלב האמצעי
**חייב** לרוץ בשרת כי הקלט שלו הוא סיסמת המשתמש:

```
app     connect → requestOneTimeLoginKey(username) → login_key
server  hash = MD5(`${login_key}|${MD5(`${user}:voximplant.com:${password}`)}`)
app     loginWithOneTimeKey(fullUsername, hash)
```

הנוסחה לעיל **verbatim מהתיעוד הרשמי** (`guides.sdk.authorization-onetimekey`).

שתי דקויות שקל לפספס, ושתיהן מקור לבאג שקט:

- ה-MD5 **הפנימי** מקבל את **שם המשתמש הקצר** (`myuser`), לא את ה-FQDN המלא.
  ה-FQDN (`user@app.account.voximplant.com`) משמש רק בקריאת ה-SDK בשלב 3.
- ה-realm הוא המחרוזת הקבועה `voximplant.com` — לא שם החשבון שלנו.

אין וריאנט שבו האפליקציה מסתדרת לבד. "שרת קודם" היא עובדה על הפרוטוקול, לא סדר עדיפויות.

## 2. מצב ההכנה בפועל

| רכיב | מצב |
|---|---|
| `console_agents.vox_username` | ✅ עמודה קיימת (מיגרציה 20260720025656) |
| ערך בפועל | ✅ `agent_1bbe74dc-…@kalfa-rsvp.kalfarsvp.voximplant.com` |
| `console_me.vox_username` | ✅ נחשף לאפליקציה |
| `enrollConsoleAgent` מקצה משתמש | ❌ כותב `user_id` + `display_name` **בלבד** |
| משתמש Voximplant מאחורי המחרוזת | ❓ **לא אומת** — ראו §3 |
| `/api/sdk-auth` | ❌ לא קיים |
| מקום לאחסון הסיסמה | ❌ לא קיים |

**הממצא המרכזי:** הערך ב-`vox_username` נכתב **מחוץ לזרימה**. שום קוד בריפו אינו
מייצר אותו ושום קוד אינו מקצה משתמש בפלטפורמה. זהו אותו דפוס שחזר לאורך כל היום —
ערך במסד שנראה מוסמך ואין לו מפיק.

## 3. השאלה שמכריעה את היקף העבודה

נבנתה פקודת אימות קריאה-בלבד:

```
npm run voximplant -- users
```

**אם המשתמש קיים** → ההכנה אמיתית; נותר §5 בלבד.
**אם לא** → נדרש גם מסלול הקצאה (§6), והיקף העבודה גדל מהותית.

## 4. תיקון לנימוק שבגללו זה נפסל

`agent-console.ts` פוסל את sdk-auth בין היתר בשל **חיוב MAU**. המחקר בריפו
(`docs/voximplant/research/getting-started.md`) קובע:

```
Active Users (MAU)   1,000 / month   בתוכנית החינמית
```

לשניים-שלושה נציגים זה אינו חסם כספי. הנימוק שנשאר תקף הוא **מערכת אימות שנייה
לתחזוקה**, לא עלות. זו הכרעת מוצר, ויש לנסח אותה ככזו.

## 5. יישום — בהנחה שהמשתמש קיים

### 5.1 אחסון הסיסמה

סוד לכל נציג. **לא** ב-`console_agents` (הטבלה נקראת דרך `console_me`).
שתי אפשרויות:

- **א׳** — `app_settings` בשדה ייעודי, כמו שאר סודות הפלטפורמה. פשוט; סוד יחיד
  משותף לכל הנציגים אם כולם חולקים משתמש אחד.
- **ב׳** — טבלה נפרדת `console_agent_secrets` עם `REVOKE ALL FROM authenticated`,
  נקראת אך ורק ב-service-role. נדרש אם לכל נציג משתמש משלו.

**המלצה:** ב׳. שם המשתמש כבר per-agent, כך שהסוד חייב להיות per-agent גם כן.

### 5.2 הנתיב

```
POST /api/agents/sdk-auth      { one_time_key }  →  { hash }
```

- אימות: `requireConsoleAgent` (Bearer). **בלי** הרשאת פלטפורמה נוספת — היכולת
  להתחבר היא עצם החברות במוקד.
- שם המשתמש **אינו מתקבל מהגוף**. הוא נקרא מ-`console_agents` לפי
  `auth.ctx.userId`. אחרת נציג א׳ יוכל לבקש hash לנציג ב׳.
- `one_time_key` — מחרוזת, אורך חסום, hex בלבד; נדחה אחרת.
- התשובה מחזירה `hash` בלבד. אף פעם לא סיסמה, לא שם משתמש, לא ה-key.
- `Cache-Control: no-store`, ו**אין** לוג של הגוף.

### 5.3 החישוב

```ts
const shortUser = voxUsername.split('@')[0];          // ה-MD5 הפנימי — שם קצר
const inner = md5(`${shortUser}:voximplant.com:${password}`);
const hash  = md5(`${oneTimeKey}|${inner}`);
```

MD5 כאן אינו בחירת אבטחה — הוא מוכתב על ידי הפרוטוקול. יש לתעד זאת בקוד כדי
שסריקת אבטחה עתידית לא "תשפר" אותו ל-SHA.

### 5.4 קצב וביקורת

- הגבלת קצב לכל נציג — בקשת hash היא ניסיון התחברות.
- לוג: הצלחה/כישלון + `staffId` בלבד. **לעולם לא** ה-key ולא ה-hash.
- לא `support_access_log` — זו אינה גישה לנכס של לקוח אלא אימות של הנציג עצמו.

### 5.5 בדיקות

וקטור ידוע מהתיעוד (user/pass/key ידועים → hash צפוי) כבדיקת רגרסיה על הנוסחה;
דחיית key זר; אי-יכולת לבקש hash עבור נציג אחר; היעדר סוד מהתשובה ומהלוגים.

## 6. יישום נוסף — אם המשתמש אינו קיים

הקצאה היא **מוטציה שמייצרת סוד**, ולכן לא ב-CLI קריאה-בלבד. פרמטרי `AddUser`
אומתו מול עץ המתודות הרשמי:

```
user_name · user_password · application_id | application_name
user_display_name · user_active · user_custom_data · parent_accounting
```

מסלול מוצע: פעולת אדמין (owner-gated) שמייצרת סיסמה אקראית, קוראת ל-`AddUser`
תחת אפליקציית `kalfa-rsvp`, שומרת את הסוד לפי §5.1, וכותבת `vox_username`
בפורמט `agent_<uuid>`. אותה פעולה צריכה גם לתקן את `enrollConsoleAgent`, שהיום
מייצר נציג בלי משתמש — מצב שנראה תקין ואינו יכול להתחבר.

## 7. מה שנשאר פתוח ואינו בשליטת הריפו הזה

- **TTL של ה-key** — התיעוד מציג כותרת "Login key TTL" אך גוף ההערה מוגש ריק ב-API
  (מגבלה מוכרת של הפלטפורמה, מתועדת ב-`docs/voximplant/README.md`). יש לאמת
  אמפירית לפני שקובעים timeout בנתיב.
- **צד האפליקציה** — `MeRow` אינו קורא `vox_username`, ואין קריאה ל-`/api/sdk-auth`.
  ריפו נפרד.
- **`monitor`/`takeover`** — sdk-auth הוא רק הכניסה. ההשתלטות עצמה דורשת Conference
  ב-VoxEngine ואימות בסביבת בדיקות; המפרט באפליקציה שגוי בנקודה הזו
  (`Conference.add()` אינו מקבל `AgentsClient`).

## 8. סדר עבודה מומלץ

1. הרצת `npm run voximplant -- users` — מכריעה בין §5 ל-§5+§6.
2. הכרעה על §5.1 (אחסון הסוד).
3. אימות אמפירי של TTL.
4. בנייה, בדיקות, ואז חיווט באפליקציה.

**אין להתחיל ב-2 לפני 1** — היקף העבודה תלוי בתוצאה.
