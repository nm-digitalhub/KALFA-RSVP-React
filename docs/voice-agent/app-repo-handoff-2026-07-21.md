# הנחיות לריפו האפליקציה (KALFA-ELEVENLABS) — 2026-07-21

> כל חוזה כאן **נשלף מהקוד החי** בריפו ה-backend ביום זה, לא מהזיכרון.
> כשמשהו לא נבנה או לא אומת — כתוב כך במפורש.

---

## 0. הכלל שמעליו הכל

**אל תציג פקד שאין מאחוריו backend.** זה החוזר הכי יקר אצלנו: כפתור "הפעל קמפיין"
לא הגיב, המשתמש עבר לאתר, השהה קמפיין ידנית כדי "לאלץ סנכרון" — והקמפיין נתקע במצב
שאי אפשר היה להחזיר ממנו. "לא נבנה" הפך ל"שבור" בידיים של המשתמש.

לכל יכולת בהמשך כתוב מפורשות אם ה-backend מוכן. מה שלא מוכן — אל תרנדר, או רנדר
מושבת עם הסבר.

---

## 1. תיקונים לבאגים קיימים

### 1.1 סטטוס קמפיין הוצג כ"מושהה" בעודו פעיל

**תוקן ב-backend.** הווי `console_campaigns` העביר עמודה מתה בשם `enabled`
(ברירת מחדל `false`, נכתבה בשום מקום). היא נגזרת עכשיו:

```sql
status = 'active' AS enabled
```

**נדרש באפליקציה:** רענון בלבד. אם המיפוי אצלכם קורא `status` ישירות — עדיף,
והוא הקנוני. `enabled` נשמר תואם-לאחור.

### 1.2 לשונית "אורחים" הראתה "אין יעדי קמפיין"

הלשונית קוראת `console_campaign_targets`, שהוא ווי מעל `outreach_state` — **יעדי
קמפיין, לא אורחי האירוע**. אירוע שקמפיינו לא הופעל מציג אפס, וזה נכון עובדתית
ומטעה למשתמש.

**נבנה לכם ווי חדש:** `console_event_guests`

```
guest_id · event_id · guest_name · dialable · phone · rsvp_status · has_active_campaign
```

- `dialable` — יש טלפון **וגם** לא ביקש הסרה.
- `phone` — מוחזר רק למי שמחזיק `view_customer_data`; אחרת `null`.
- `has_active_campaign` — משקף את אותו גייט שה-route של החיוג בודק, כדי שלא תציגו
  כפתור שייענה ב-409.

**נדרש באפליקציה:** להחליף את מקור הלשונית ל-`console_event_guests`, או לשנות את
הכותרת ל"יעדי קמפיין". אל תשאירו "אורחים" מעל יעדים.

### 1.3 מחרוזת סטטוס לא מתורגמת

בכרטיס אורח מוצג `exhausted` באנגלית בממשק עברי.

### 1.4 אירוע שנמחק נשאר מוצג

אירוע שנמחק מהשרת המשיך להופיע באפליקציה עם סטטיסטיקות, שעה אחרי המחיקה. אין
טיפול בביטול ישות. **נדרש:** ריענון מלא של הרשימה, או invalidation ב-404.

## 2. מה שה-backend מוכן אליו — חוזה מדויק

כל הנתיבים: `POST`, בסיס `https://beta.kalfa.me`, כותרת
`Authorization: Bearer <supabase-jwt>`.

| נתיב | הרשאה נדרשת | מוכן |
|---|---|---|
| `/api/agents/status` | — (חברות במוקד) | ✅ |
| `/api/calls/{id}/agent-command` | `manage_voice` | ✅ |
| `/api/calls/{id}/end` | `manage_voice` | ✅ |
| `/api/events/{eventId}/outreach-call` | `manage_voice` | ✅ |
| `/api/campaigns/{id}/status` | **`campaigns.runstate`** | ✅ חדש |

ההרשאות מגיעות ב-`console_me.permissions` (מערך מפתחות). **גדרו את ה-UI לפיהן** —
אל תציגו פקד שהמשתמש יקבל עליו 403.

### 2.1 השהיה והחייה של קמפיין — חדש

```
POST /api/campaigns/{id}/status
{ "action": "activate" | "pause" }
→ 200 { ok: true, status: "paused" | "active" }
```

**הרשאה: `campaigns.runstate`** — לא `manage_voice`. מפתח חדש שנוסף היום.

**ההיקף מצומצם בכוונה, וזו החלטת בעלים:**

```
active  → paused    מותר לצוות
paused  → active    מותר לצוות   ← החייה בלבד
approved/scheduled → active    ❌ 409 — הפעלה ראשונה היא של בעל האירוע באתר
```

**אל תציגו "הפעל קמפיין" על קמפיין שמעולם לא הופעל.** הוא ייענה ב-409. הצג את
הפקד רק כאשר `status === 'paused'`.

**קודי תשובה:**

| קוד | משמעות | מה להציג |
|---|---|---|
| 200 | בוצע | הסטטוס החדש |
| 401 | לא נציג מוקד | להתנתק |
| 403 | אין `campaigns.runstate` | להסתיר את הפקד מלכתחילה |
| 409 | המצב אינו מתיר | **הצג את `error` כמו שהוא** — הוא עברית קריאה |
| 500 | תקלה אצלנו | הודעה גנרית |

ה-409 מחזיר טקסט אמיתי: *"להפעלת הקמפיין נדרשת תפיסת מסגרת מאושרת"*, *"האירוע כבר
חלף"*, *"יש לפרסם את האירוע"*. אל תחליפו אותו בהודעה כללית — הוא הסיבה.

### 2.2 לחישה לסוכן ה-AI — **מוכן, ולא דורש SDK**

זו הנקודה שהכי כדאי שתדעו: **היכולת שנשמעת הכי מתקדמת כבר עובדת.**

```
POST /api/calls/{callAttemptId}/agent-command
{ "command": "contextual_update", "text": "..." }   לחישה שלא קוטעת
{ "command": "user_message",      "text": "..." }   מזריק תור, קוטע
{ "command": "clear_buffer" }                        barge-in
{ "command": "close_agent" }                         סוגר את רגל ה-AI
→ 202 { delivered: true, applied: "pending", command, request_id }
```

הגוף **שטוח** (`text` ברמה העליונה, לא תחת `payload`) — כפי שהאפליקציה כבר שולחת.

**אין צורך בהתחברות ל-SDK, אין משתמש ב-Voximplant, אין MAU.** הנציג מנחה את ה-AI
בטקסט; האורח לא שומע דבר.

**קריטי בתצוגה — `202` אינו "בוצע":**

- `delivered: true` = הפקודה הגיעה לסשן החי.
- `applied: "pending"` = **לא ידוע** אם המודל פעל לפיה. עבור `contextual_update`
  ו-`user_message` זה יישאר `pending` **לתמיד** — ElevenLabs לא מחזיר אישור.

אל תציגו "הסוכן ביצע". הניסוח הנכון: *"נשלח לסוכן"*.

**כל פקודה נרשמת** מהיום ב-`console_agent_commands`: מי, מתי, איזו שיחה, מה נאמר,
האם נמסר — כולל ניסיונות שנכשלו. זה רלוונטי לכם משום שהמשתמש צריך לדעת שההתערבות
מתועדת.

### 2.3 חיוג יזום

```
POST /api/events/{eventId}/outreach-call
{ "guest_id": "<uuid>" }
→ 202 { status: "accepted", dispatch_id }
```

**שני תיקונים נדרשים בצד שלכם** (מתועדים כליקויים קיימים):

1. `event_id` נשלח כמחרוזת קשיחה `"default-event"`. חייב להיות UUID אמיתי — ה-route
   דוחה אחרת.
2. ה-JSON נבנה בשרשור מחרוזות עם הטלפון בפנים. השתמשו בסריאלייזר.

**`202` = נכנס לתור, לא "מחייג".** כל הגייטים (הסכמה, DNC, כבר-נוצר-קשר, אירוע
פעיל, מסגרת) נבדקים **בעובד**, אחרי התשובה. שיחה עשויה להידחות בשקט אחריה.
הציגו "הבקשה נקלטה", ואל תבטיחו שיחה.

## 3. מה ש**לא** מוכן — אל תבנו מולו

### 3.1 האזנה והשתלטות — חסום

```
/api/calls/{id}/monitor     ❌ לא קיים
/api/calls/{id}/takeover    ❌ לא קיים
```

**שלוש סיבות, כולן אמיתיות:**

1. **אין משתמש ב-Voximplant.** אומת היום: אפס משתמשים בשתי האפליקציות
   (`kalfa-rsvp`, `kalfatest`). הערך ב-`console_me.vox_username` הוא **מחרוזת
   יתומה** — שום קוד לא מייצר אותה ואין משתמש מאחוריה.
2. **`/api/sdk-auth` לא קיים**, ובלעדיו התחברות ה-SDK בלתי אפשרית: הפרוטוקול דורש
   שהשרת יחשב hash מסיסמה שלעולם לא תשב ב-APK.
3. **המפרט שלכם שגוי בנקודה הזו:** `Conference.add()` אינו מקבל `AgentsClient`.
   בנייה לפיו תייצר קוד שמתקמפל ולא עובד.

**נדרש מכם עכשיו:** אל תקראו ל-`vox_username`, ואל תנסו להתחבר ל-SDK. אם יש כפתור
"האזן"/"השתלט" — הסתירו או השביתו עם הסבר.

**כשזה ייבנה** — הזהות תהיה **פר-נציג**, לא משותפת: `VoxEngine.callUser({username})`
מכניס אדם לשיחה לפי שם המשתמש, כך שאי אפשר לכוון לנציג שתפס שיחה בזהות משותפת.

### 3.2 סחף סכמה שלא נלכד

`console_call_feed` קיבל ב-20.7 שלוש עמודות שה-DTO שלכם לא קורא:

```
takeover_claimed_at · takeover_request_id · participation_state
```

אלה שדות התיאום שמונעים ששני נציגים יתפסו את אותה שיחה. **כל מימוש השתלטות חייב
להשתמש בהם** — היום האפליקציה הייתה מתחרה על אותה שיחה.

### 3.3 `saveRsvpResult` ריקה

מסך השיחה אוסף תשובה, מספר אורחים והערות, קורא לפונקציה — **ולא כותב כלום**, ואז
מנתק. אין route לכתיבת RSVP מהקונסולה, ולא יהיה: תוצאות RSVP שייכות לצינור
ה-client-tools של ElevenLabs.

**נדרש:** להשבית את הטופס, לא להעמיד פנים שהוא שומר.

## 4. מה שהאפליקציה כותבת ישירות (בלי route)

מותר, וההרשאות ניתנו:

- `agent_status` — עדכון השורה **של הנציג עצמו**.
- `console_call_feed.handled_by` / `agent_id` — בעלות על השתלטות בלבד.

כל השאר קריאה בלבד. **מצב קמפיין לעולם לא נהפך מלקוח** — הוא צמוד לחיוב ב-SUMIT
ועובר אך ורק דרך `/api/campaigns/{id}/status`.

## 5. תיאור הנציג בממשק — לא מדויק

הטקסט הנוכחי מבטיח *"מאזין לשיחות... ויכול להשתלט עליהן ולדבר עם האורח בעצמו"*.
**שתי היכולות שהוא מדגיש הן בדיוק השתיים שלא קיימות.**

מה שנציג מוקד באמת מקבל היום:

- גישת צוות **חוצת-לקוחות** לכל נתוני המוקד — אירועים, אורחים, תוצאות RSVP, ניתוחי
  שיחות של כל בעלי האירועים בפלטפורמה
- התערבות בשיחה חיה בטקסט (לחישה/הזרקה/ניתוק)
- חיוג יזום
- השהיה והחייה של קמפיין

זה משמעותי — פשוט לא מה שכתוב. **נדרש:** לנסח מחדש לפי הקיים, ולציין שהאזנה
והשתלטות בפיתוח.
